## Middleware Outcome Summary

- If Google auth is not requested, HTTP middleware lets requests through and route-local checks decide behavior.
- If Google auth is requested and the request is loopback/local-trusted, middleware bypasses Google auth and route-local checks decide behavior.
- Explicit middleware exemptions are `/`, `/logged-out`, `/health`, `/health/detailed`, `/api/email-inbound`, `/auth/google/login`, `/auth/google/callback`, `/auth/device/google`, `/auth/logout`, `/auth/session`, `/client/bootstrap`, `/apk`, and `/apps/*`.
- If Google auth is requested but incomplete, non-exempt public HTTP paths return 503 JSON `{"detail":"Google auth is enabled but incomplete"}`.
- On a public host with ready Google auth, non-exempt paths accept a valid bearer device token or signed browser session. Unauthenticated `/watch*` redirects 302 to Google login; other unauthenticated non-exempt paths return 401 JSON with `detail` and `login_url`.
- Route-local secret/device checks remain authoritative after middleware. Important examples are `x-sm-hook-secret` for remote-session hooks, email worker secret/sender checks for inbound email, node-agent secret in the WebSocket hello frame, and mobile terminal device-key signatures.
