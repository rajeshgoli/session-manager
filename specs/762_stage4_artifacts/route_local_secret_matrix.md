# Stage 4 Route-Local Secret Matrix

Status: converged after three sequential independent reviewer convergence signals; owner security feedback incorporated after convergence.

This matrix satisfies the Stage 4 requirement that route-local secret mechanisms state current missing, mismatch, reuse, logging, and rotation behavior. The current-behavior columns are source evidence. Stage 5 now approves selected cutover changes for the Rust port: proof-required public access, explicit device/node revocation, non-loopback hook proof, retained but stricter email worker proof, removal of Telegram surfaces, rejection of node hook-secret fallback, and auth/proof or signing for app artifacts.

| Mechanism | Current Missing Behavior | Current Mismatch / Invalid Behavior | Reuse / Replay Behavior | Logging / Redaction Requirement | Rotation / Stage 5 Gate |
| --- | --- | --- | --- | --- | --- |
| browser OAuth `state` | Missing callback `code` or `state` redirects to `/watch/?auth_error=missing_code`. Missing stored session state redirects as invalid state and clears session. | State mismatch clears session and redirects to `/watch/?auth_error=invalid_state`. OAuth denial redirects with `google_denied`; exchange/userinfo failure redirects with `exchange_failed`; unverified or unallowlisted email redirects with `unauthorized_email`. | One OAuth state is stored per browser session. Successful callback clears prior session state before writing authenticated fields. OAuth code reuse is delegated to Google. | Do not log code, access token, client secret, signed cookie, or OAuth state. Existing rejected-login log may include email and verified flag only. | Cookie secret comes from config/local env and may be derived from Google web client secret. Rotating it invalidates browser sessions and device bearer tokens; Stage 5 permits this only as an explicit cutover re-login/re-enrollment gate recorded in the ledger. |
| signed browser session cookie `sm_auth` | Unauthenticated public non-exempt paths return JSON 401 or redirect `/watch` to login depending on path; `/auth/session` reports current auth mode. | Bad/expired/unsignable cookie is treated as unauthenticated by middleware/session layer. | Cookie remains valid while signed session data is accepted by middleware; no Stage 2/3 explicit max age was recorded. | Do not log cookie value or signing secret. Logs may report auth mode/failure counts only. | Same as OAuth cookie secret. Expiry/rotation is allowed when implemented as an explicit cutover re-login/re-enrollment gate. |
| native device bearer token `smat_...` | Missing bearer on public non-exempt route falls back to browser session or unauthenticated behavior. Missing signing secret makes device-token issuing unavailable. | Bad HMAC, malformed token, wrong type, or expired `exp` is rejected as unauthenticated. `/auth/device/google` returns 401 for invalid Google ID token and wrong audience, 403 `Google account is not allowlisted` for unverified or non-allowlisted email, and 503 for Google auth/device-signing misconfiguration. | Bearer token is reusable until expiry or signing-secret rotation. No per-token revocation store is recorded in Stage 2/3. | Do not log bearer token, payload signature, Google ID token, or signing secret. Auth failure logs must be privacy-safe and preserve the 403 allowlist/unverified distinction. | Signing-secret rotation invalidates all outstanding tokens. Device inventory/revocation and shorter-lived proofed access are accepted Stage 5 requirements; rollout must force re-login/re-enrollment explicitly rather than silently failing. |
| mobile terminal device-key signature | Missing any attach-ticket proof header `x-sm-device-key-id`, `x-sm-device-timestamp`, `x-sm-device-nonce`, or `x-sm-device-signature` returns HTTP 401 `Device key proof is required`. WebSocket auth-frame missing ticket/device/nonce/signature fields returns error frame `Invalid terminal auth frame`; the shared signature helper also preserves 401 `Missing device signature` if reached with an empty signature. | Attach-ticket proof preserves exact HTTP details: 401 `Invalid device timestamp`, 401 `Expired device signature`, 401 `Invalid device signature encoding`, 401 `Invalid device signature`, 401 `Invalid device public key`, 401 `Unsupported device public key type`, and 401 `Device key is not registered`. User/session gates remain distinct: 403 `User is not allowed to attach`, 403 `Interactive shell access is not enabled for this user`, 403 attach-not-supported reason, 409 `Session is not running`; WebSocket revalidation preserves 403 `User is no longer allowed to attach`, 401 `Device key is no longer registered`, and protocol-manifest ticket errors. | Timestamp skew is bounded; no persistent nonce replay cache is recorded. WebSocket ticket consume is single-use, limiting replay after ticket issuance. | Do not log raw signature, public key material beyond configured diagnostics, or device private material. Audit may include user/device key id and failure reason. Fixture exact HTTP details and WebSocket error-frame messages without logging proof material. | Device-key revocation is required for proofed public mobile access. Persistent nonce replay cache or mandatory key rotation is accepted when fixture-locked and surfaced through re-enrollment/revocation UX. |
| mobile attach ticket secret | Missing ticket id/secret/device/nonce/signature in WebSocket auth frame closes with policy error. | Device mismatch, secret mismatch, expired/consumed ticket, quota failure, revoked user/device, or unattached session sends current error frame/1008 behavior from protocol manifest. | Ticket is process-local, short-lived, replaced per pending user/session/device tuple, and consumed once under lock. Restart invalidates outstanding tickets. | Do not log ticket secret. Audit may include ticket id, user, session, device key id, and failure reason. | Durable tickets are not current behavior. Rust may keep tickets or replace them with direct signed WebSocket attach proof, but must preserve user auth, device proof, session binding, quotas, revocation, audit, and fail-closed behavior. |
| remote hook secret `x-sm-hook-secret` | For remote sessions with configured hook secret, missing header is equivalent to mismatch and returns 403 `Invalid hook secret`. If no hook secret is configured for that remote session, current source allows the hook after middleware. | Mismatch returns 403 `Invalid hook secret` using constant-time comparison. | Static shared secret is reusable until config changes. No nonce/replay protection is recorded. | Do not log header value or configured hook secret. Logs may identify session/node and failure class. | Stage 5 cutover requires route-local proof/secret for all non-loopback hooks. No-secret remote configs fail closed or become local-only. |
| node-agent hello secret | Missing configured secret for node sends error `Node-agent secret not configured` and closes 1008. Missing/invalid first `hello` closes per protocol. | Wrong supplied secret sends `Invalid node-agent secret` and closes 1008; `primary` or empty node id is rejected. | Static `node_token` or hook-secret fallback is reusable for each node-agent connection. | Do not log supplied secret, node token, or hook secret. Logs may include node id and failure type. | Stage 5 rejects hook-secret fallback. Registered nodes need explicit node credentials, rotation/removal UX, and public-edge node proof for public fallback. |
| inbound email worker secret | If worker secret is configured, missing header returns 401 `Invalid email worker secret`. If no worker secret is configured, current route-local secret check is skipped, but trusted session-id header must not be trusted without worker secret. | Mismatch returns 401 `Invalid email worker secret`. Unauthorized sender and missing routing content are handled by email bridge behavior. | Static worker secret is reusable by the email worker. No nonce/replay protection is recorded. | Do not log worker secret or full raw email in auth failure paths. Logs/audit may include sender, route, accept/reject reason, and target session when resolved. | Email/human delivery and inbound email are retained. Stage 5 requires worker proof before accepting public inbound email or trusting the session-id header, sender allowlist enforcement, fixed/default or explicitly allowlisted webhook routes, and privacy-safe audit. |
| app upload actor auth | Upload requires actor email from device/cookie/local bypass when Google auth is requested; missing actor fails current route behavior. | Unauthenticated public upload follows middleware 401/redirect behavior; invalid app/form/version/APK size returns route-specific errors. | Authenticated actor can upload repeatedly; immutable hash files are copied only if absent and latest/meta are atomically replaced. | Do not log bearer/cookie values. Audit/log upload actor, app, hash, size, and validation failure reason. | Stage 5 requires auth/proof or artifact signing before public artifact serving. Public unauthenticated app artifact serving is not a Rust target. |
| queue/review dedupe tokens | Dedupe tokens are not authentication secrets. Missing token means no dedupe behavior for the affected queue/policy path. | Mismatch means a distinct dedupe key; it must not grant authority. | Reuse intentionally deduplicates or locates prior work depending on the endpoint. | Treat as metadata, not credential. Avoid logging if it can contain operator-provided sensitive text. | Changing dedupe semantics can affect queue/review behavior and belongs in Stage 5 if externally observable. |
| public-edge device/node proof and edge assertion | Not current Python behavior. Rust public remote access must fail closed when phone proof, node proof, or edge assertion is missing. Origin must reject public forwarded traffic without a valid edge assertion even if Cloudflare forwards it. | Invalid proof, revoked device/node id, wrong route binding, wrong node id, bad signature/MAC, bad controller assertion, or route not on the edge allowlist must deny without returning operational data. | Proof must be nonce/challenge or timestamp-window bound, method/path/body-bound where applicable, and replay-protected at edge and origin. WebSocket proof must bind the upgrade/session to the device/node identity. | Do not log private key material, raw proof signatures, bearer tokens, controller IPC token, edge signing secret, or full request bodies. Logs may include device/node id, route, failure class, and revocation status. | Required gates: `sm list-devices`, `sm remove-device <id>` or equivalent, node token rotation/removal, edge-signing/controller-token rotation, downgrade behavior, and black-box fail-closed fixtures from `public_edge_proof_model.md`. |
